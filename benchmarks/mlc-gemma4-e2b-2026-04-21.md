## Compile Command

```bash
sudo docker run --rm --runtime nvidia \
  -v /opt/models/gemma4-e2b-q4f16-mlc:/weights \
  rimrock/mlc-gemma4:latest \
  python3 -m mlc_llm compile /weights \
    --device cuda \
    --opt O3 \
    --output /weights/lib-sm87.so
```

Output: `/opt/models/gemma4-e2b-q4f16-mlc/lib-sm87.so` (38MB)

## Notes

- FlashInfer `batch_prefill` JIT op fails to load (missing `.sources` attr) — model falls back to TIR-based KV cache. This is a known limitation of the PR branch.

- `mlc_llm bench` subcommand not present in this PR branch; benchmark used `MLCEngine` Python API directly.

## Text Quality Benchmark (v3, 2026-04-21 — Round 1: old weights + wrong chat format)

Ran `text_benchmark_v3` against the MLC engine. **Average score: 1.0/5 across all 10 questions.**

Root causes: (1) welcoma pre-quantized weights used wrong RoPE interleaving (GPT-J) and phantom k/v weights for shared-KV layers; (2) chat template used `<start_of_turn>user` (Gemma 2 format) instead of `<|turn>user` (Gemma 4 native format).

## Re-quantization + Compile (2026-04-22)

Full re-quantization pipeline completed:
- Codex patches applied to source: removed GPT-J RoPE forcing, made k_proj/v_proj conditional on `shared_cache_source is None`
- `mlc_llm convert_weight` from `google/gemma-4-e2b-it` HF weights → **45 shards, 2.444 GB** (vs 46 shards 2.512 GB with phantom weights — 68 MB difference confirms phantom weights removed)
- `mlc_llm compile --opt O3 --device cuda` → `lib-sm87-fresh.so` (35 MB)
- Fixed `mlc-chat-config.json`: chat tokens corrected to Gemma 4 native (`<|turn>` / `<turn|>`), context 4096, prefill 1024, repetition_penalty 1.15

**Tokenizer special tokens (Gemma 4):**
- Token 105 = `<|turn>` (start of turn)
- Token 106 = `<turn|>` (end of turn / stop token)

**Conclusion:** Re-quantization + correct chat format raised quality from 1.0/5 to 2.8/5. Throughput confirmed at 33.3 tok/s (beats 26–29 tok/s baseline). The 2B model handles factual recall and practical tasks well but struggles with code generation, arithmetic, and multi-turn context. Suitable for use cases not requiring precise computation or long multi-turn conversations.
