## Compile Command

```bash
sudo docker run --rm --runtime nvidia \
  -v /opt/models/gemma4-e2b-q4f16-mlc:/weights \
  rimrock/mlc-gemma4:latest \
  python3 -m mlc_llm compile /weights \
    --device cuda \
    --opt O3 \
    --output /weights/lib-sm87.so
```

Output: `/opt/models/gemma4-e2b-q4f16-mlc/lib-sm87.so` (38MB)

- `mlc_llm bench` subcommand not present in this PR branch; benchmark used `MLCEngine` Python API directly.

## Text Quality Benchmark (v3, 2026-04-21)

Ran `text_benchmark_v3` against the MLC engine. **Average score: 1.0/5 across all 10 questions.**

| Symptom | Affected questions | Root cause |
|---|---|---|
| 1 token / empty response | Q2, Q3, Q4, Q5 | Model immediately generates stop token; likely KV cache or attention bug in draft PR |
| Repetition collapse (hits max_tokens) | Q6, Q7 | Degenerate greedy decoding; `repetition_penalty` in `mlc-chat-config.json` is not applied at runtime |
| Truncated / partial answer | Q1, Q9, Q10 | Model hallucinates topic but loses instruction format |
| Multi-turn context failure | Q8 | Sliding window KV cache not preserving turn context across completions |

Throughput for questions that do generate (Q6, Q7, Q9, Q10): **32–36 tok/s** — GPU inference confirmed working.

**Conclusion:** The draft PR (`mlc-ai/mlc-llm #3485`) has verified throughput (33.1 tok/s, beats baseline) but unverified quality. The Gemma 4 novel attention architecture (alternating sliding/full attention, KV sharing across layers) likely has numerical or implementation bugs that produce wrong outputs despite correct token generation speed. Do not use for production inference until PRs are merged and validated.
